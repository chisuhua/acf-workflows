### 慢循环执行过程（45 分钟）

#### Step 1: 架构需求分析（5 分钟）

**输入**:
```
用户需求：电商数据项目是作为 ZCF 双循环工作流的验证试验

阶段 2 目标：
- 实现分析模块（分类/情感/价格趋势）
- 验证 Skill + Agent 并行执行架构
```

**架构问题**:
1. 分析功能如何封装？→ 决定：每个分析功能封装为 Skill
2. 如何支持并行执行？→ 决定：Orchestrator + Analyst Agents 模式
3. 结果如何汇总？→ 决定：Reporter Agent 统一生成报告

---

#### Step 2: 架构方案设计（15 分钟）

**设计输出**:

```markdown
# 阶段 2 架构设计（Skill + Agent 协同）

## 分层架构
用户请求 → Orchestrator（编排 Agent）
              ↓
    并行 spawn 3 个 Analysts
    ├─ Analyst-1: classifier-skill
    ├─ Analyst-2: sentiment-skill
    └─ Analyst-3: price-trend-skill
              ↓
         Reporter（汇总生成报告）

## Skills（工具封装层）
- ecommerce-crawler → src/crawler/main.py
- ecommerce-classifier → src/analysis/classifier.py
- ecommerce-sentiment → src/analysis/sentiment.py
- ecommerce-price-trend → src/analysis/price_trend.py

## Agents（并行协调层）
- orchestrator → 解析意图 + 分派任务
- analyst-1/2/3 → 调用对应 Skill
- reporter → 汇总结果生成报告
```

**关键决策**:
- ✅ Skill 作为薄封装层，核心逻辑在 src/
- ✅ Agents 使用 OpenClaw 的 sessions_spawn 并行执行
- ✅ 报告格式统一为 Markdown

## 🔍 关键决策点复盘

### 决策点 1: 阶段 2 启动慢循环

**时机**: 阶段 1 完成后

**选项**:
- A: 直接编码（风险：架构不清晰，可能返工）
- B: 启动慢循环（成本：30 分钟架构设计）

**决策**: 选项 B

- 阶段 2 涉及 Skill+Agent 新架构，无先例可循

**结果**: ✅ 正确决策，阶段 2 编码仅用 15 分钟（无返工）

**时机**: 阶段 2 完成后

- A: 启动慢循环（设计集成测试架构）
- B: 直接编码（集成测试架构明确）

**决策**: 选项 B

**理由**:
- 集成测试架构明确（单元测试 + 基准测试 + 错误处理）

```

  └─ 否 → 继续下一题
```

## 🎉 核心学习

### 1. 慢循环的价值

**学习**: 慢循环不是"浪费时间"，而是**投资**

**证据**:
- 阶段 2 投入 30 分钟架构设计
- 编码仅用 15 分钟（无返工）
- 如果跳过慢循环，预估返工时间 >60 分钟

**ROI**: 30 分钟投入 → 节省 60+ 分钟返工 = **2x 回报**

---

### 2. 何时启动慢循环

**学习**: 不是每个阶段都需要慢循环

**规则**:
- ✅ 架构未明确 → 启动慢循环
- ✅ 方案有选择 → 启动慢循环
- ✅ 有技术风险 → 启动慢循环
- ❌ 架构明确 → 直接快循环
- ❌ 标准流程 → 直接快循环

---

### 3. 快循环自动流转的关键

**学习**: 快循环无需老板决策，前提是：
1. 架构明确（慢循环已完成）
2. 任务分解清晰（implementation-plan.md）
3. 评审标准明确（DoD）

**证据**: 电商项目 99 个测试用例，全程自动流转，仅阶段 2 启动 1 次慢循环
