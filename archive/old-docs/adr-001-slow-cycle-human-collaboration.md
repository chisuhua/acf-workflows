# ADR-001: 慢循环必须 DevMate + 老板协同

**日期**: 2026-03-29  
**状态**: ✅ 已采纳  
**项目**: ACF 工作流

---

## 背景

在 Agent News 项目的架构循环实践中，发现以下问题：

1. **自动化工具的局限**: `/zcf:arch-doc` 等 OpenCode 命令适合按模板生成文档，但不适合架构讨论
2. **架构决策需要协同**: 架构设计涉及多方案对比、技术选型、权衡取舍，需要 DevMate + 老板共同决策
3. **手动 vs 自动的边界**: 需要明确哪些步骤应该手动（架构讨论），哪些可以自动（文档生成）

---

## 决策

**慢循环（架构循环）必须 DevMate + 老板协同，快循环使用/zcf:arch-doc 生成详细设计，最终架构文档必须老板最终意见确认**

### 慢循环范围

| 活动 | 执行方式 | 工具 | 产出物 |
|------|---------|------|--------|
| 架构需求分析 | DevMate + 老板 | 对话讨论 | 需求文档 |
| 架构方案设计 | DevMate 手动创建 | `write` 工具 | 系统架构设计 |
| 多方案对比 | DevMate + 老板 | 对话决策 | 技术选型记录 |
| 架构评审 | DevMate + 老板 | 对话确认 | 评审报告 |
| 同步到编码仓库 | `skill_use acf-sync` | ACF Skill | 编码仓库架构文档 |

### 快循环范围

| 活动 | 执行方式 | 工具 | 产出物 |
|------|---------|------|--------|
| 模块详细设计 | OpenCode | `/zcf:arch-doc` | detailed-design.md |
| API 规范生成 | OpenCode | `/zcf:arch-doc` | api-spec.md |
| 数据库设计 | OpenCode | `/zcf:arch-doc` | database-schema.md |
| 测试用例生成 | OpenCode | `/zcf:task-review` | 测试报告 |
| 代码实现 | OpenCode | `task(...)` | 源代码 |
| 单元测试 | OpenCode | `pytest` | 测试通过报告 |

### 文档流转关系

```
慢循环（提案仓库）          快循环（编码仓库）
├── 系统架构设计     ─────→  ├── system-architecture.md (同步)
├── 模块设计方案     ─────→  └── phases/phase-1/
└── 技术选型决策            ├── crawler/detailed-design.md (生成)
                            ├── processor/detailed-design.md (生成)
                            └── storage/detailed-design.md (生成)
                                 ↓
                            对比评估（DevMate）
                                 ↓
                            综合建议方案
                                 ↓
                            老板最终意见 ⭐
                                 ↓
                            最终架构文档（权威版本）
```

**慢循环文档** 约束 **快循环文档**，**老板意见** 决定 **最终文档**

## 决策理由

### 1. 架构决策需要人类判断

**架构设计涉及**:
- 技术选型（SQLite vs MySQL）
- 架构风格（分层 vs 微服务）
- 权衡取舍（性能 vs 可维护性）
- 风险评估（技术风险、进度风险）

**这些无法自动化**:
- 需要业务理解
- 需要经验判断
- 需要风险偏好评估

---

### 2. 慢循环的本质是"思考"，不是"执行"

**慢循环特点**:
- 频率：按需触发
- 耗时：21 分钟（本次实践）
- 产出：架构决策、设计方案
- 参与者：DevMate + 老板

**快循环特点**:
- 频率：每日执行
- 耗时：60 分钟/Task
- 产出：代码、测试
- 参与者：OpenCode/Claude Code

**慢循环不适合自动化**:
- 思考过程无法自动化
- 决策过程需要人类判断
- 讨论过程需要双向沟通

---

### 3. /zcf:arch-doc 等工具的定位

**适合场景**:
- 按模板生成详细设计文档
- 生成 API 规范
- 生成测试策略文档

**不适合场景**:
- 架构方案讨论
- 技术选型决策
- 多方案对比

**正确用法**:
```
慢循环（DevMate + 老板）
    ↓ 架构决策完成
快循环（OpenCode）
    ↓
/zcf:arch-doc "阶段 1：爬虫模块详细设计"
    ↓
按模板生成标准化文档
```

---

## 后果

### 正面后果 ✅

1. **架构质量提升**: DevMate + 老板协同，避免个人偏见
2. **决策透明**: 讨论过程记录在架构文档中
3. **责任明确**: 架构决策由 DevMate + 老板共同负责
4. **工具定位清晰**: /zcf:arch-doc 用于文档生成，不用于架构讨论

### 负面后果 ⚠️

1. **时间成本**: 慢循环需要 DevMate + 老板时间投入（本次 21 分钟）
2. **无法完全自动化**: 架构循环无法实现端到端自动化

### 权衡

**时间投入 vs 架构质量**:
- 21 分钟架构讨论 → 避免数小时返工
- 人工决策 → 保证架构符合业务需求

## 实施指南

### 慢循环执行步骤

1. **DevMate 手动创建架构文档**
   ```
   不使用 /zcf:arch-doc（架构讨论需要灵活性）
   使用 write 工具直接创建
   产出：系统架构设计、模块设计方案
   ```

2. **DevMate + 老板讨论架构方案**
   ```
   多方案对比
   技术选型决策
   风险评估
   ```

3. **架构评审**
   ```
   DevMate 创建评审报告
   老板确认评审结论
   ```

4. **同步到编码仓库**
   ```bash
   skill_use acf-sync
   # 仅同步系统架构和模块设计方案
   ```

### 快循环执行步骤

1. **使用 /zcf:arch-doc 生成详细设计**
   ```bash
   # 基于慢循环的设计方案，生成标准化详细设计
   /zcf:arch-doc "阶段 1：爬虫模块详细设计"
   /zcf:arch-doc "阶段 1：处理模块详细设计"
   /zcf:arch-doc "阶段 1：存储模块详细设计"
   ```

2. **使用 /zcf:task-review 进行任务评审**
   ```bash
   /zcf:task-review "Task 001 完成"
   ```

### 架构调整流程

#### 小调整（接口参数、实现细节）

```bash
# 快循环中直接更新
/zcf:arch-doc "更新爬虫模块 API 规范"
# 无需老板确认
```

#### 中等调整（模块接口、新增功能）

```bash
# 1. DevMate 在提案仓库更新设计方案
# 2. 老板确认 ⭐
# 3. 同步到编码仓库
skill_use acf-sync
# 4. 重新生成详细设计
/zcf:arch-doc "阶段 1：爬虫模块详细设计"
```

#### 重大调整（系统架构、技术选型）

```bash
# 1. 暂停快循环
# 2. 返回慢循环（架构讨论）
# 3. DevMate + 老板讨论新方案 ⭐
# 4. 更新系统架构设计（手动）
# 5. 老板评审通过 ⭐
# 6. 同步到编码仓库
skill_use acf-sync
# 7. 恢复快循环
# 8. 重新生成所有详细设计
/zcf:arch-doc "阶段 1：爬虫模块详细设计"
/zcf:arch-doc "阶段 1：处理模块详细设计"
/zcf:arch-doc "阶段 1：存储模块详细设计"
```

### 最终架构文档确认流程

```bash
# 1. DevMate 对比评估
# 填写 design-comparison-template.md
# 产出：综合建议方案

# 2. 老板最终意见 ⭐
# Review 综合建议方案
# 决策：✅ 通过 / ⚠️ 调整 / ❌ 重新设计

# 3. 生成最终架构文档
# 基于老板确认的方案
# 作为编码实施的权威依据

# 4. 开始编码 Task 001
task(prompt="Task 001: 创建爬虫基础结构")
```
