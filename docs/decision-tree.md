## 1. 任务接收决策树（强制流程）

```
收到任务
    │
    ▼
┌─────────────────────────────────────┐
│ 步骤 1: 需求澄清检查                 │
│ 问自己以下 4 个问题：                │
│ □ 交付物格式明确？（代码/文档/图片） │
│ □ 技术选型/偏好明确？               │
│ □ 优先级明确？（性能/可读性/速度）  │
│ □ 范围边界明确？（验收标准）        │
└─────────────────────────────────────┘
    │
    ├─ 有任一模糊 ──→ 🛑 触发 Interview（最多 5 问，2 轮上限）
    │                  格式：选择题 + 1 个开放题
    │                  输出：写入 memory/YYYY-MM-DD.md
    │
    ▼ 全部明确
┌─────────────────────────────────────┐
│ 步骤 2: 架构复杂度评估              │
│ 问自己以下 4 个问题：                │
│ □ 是否涉及新领域/新技术栈？         │
│ □ 是否影响现有系统架构？            │
│ □ 是否有重大技术选型决策？          │
│ □ 预估工作量 > 1 天？               │
└─────────────────────────────────────┘
    │
    ├─ 全部否 ──→ 进入编码循环（快循环）
    │              直接调用 acf-executor 执行
    │
    ▼ 有任一是
┌─────────────────────────────────────┐
│ 步骤 3: 架构循环（慢循环）           │
│ 3.1 DevMate 手工创建架构草稿         │
│     → /workspace/mynotes/<Project>/ │
│       docs/architecture/draft.md    │
│ 3.2 找老板评审草稿（关键决策点）     │
│     - 技术选型                      │
│     - 模块边界                      │
│     - 数据流向                      │
│ 3.3 老板确认后 → 派给 OpenCode       │
│     /zcf/arch-doc "<主题>"          │
│ 3.4 对比评审（mynotes vs 编码仓库）  │
│     - 一致 → 架构定稿，进入步骤 3.5   │
│     - 不一致 → 返回步骤 3.1          │
│ 3.5 acf-sync 同步 → 进入编码循环     │
└─────────────────────────────────────┘
```

## 2. 架构循环 ↔ 编码循环 切换规则

### 2.1 进入架构循环的条件（满足任一即切换）

| 条件 | 严重性 | 行动 |
|------|--------|------|
| ❌ 架构缺陷（严重） | 严重 | 暂停编码，返回架构循环 |
| ❌ 技术选型变更 | 严重 | 暂停编码，返回架构循环 |
| ❌ 需求重大变更（>30%） | 严重 | 暂停编码，返回架构循环 |
| ❌ 新增模块 | 中等 | 评估后决定是否返回架构循环 |
| ❌ 老板决策 | 严重 | 立即返回架构循环 |
| ❌ 架构违规（评审发现） | 严重 | 暂停任务，返回架构循环 |
| ❌ 依赖方向错误（评审发现） | 严重 | 暂停任务，返回架构循环 |
| ❌ 评审发现 P0 问题 | 严重 | 暂停后续任务，返回架构循环 |

### 2.2 保持编码循环的条件（满足全部即继续）

| 条件 | 说明 |
|------|------|
| ✅ 任务在已定稿架构范围内 | 无需架构调整 |
| ✅ 技术选型无变更 | 沿用已有决策 |
| ✅ 偏差为轻微级别 | 记录 CHANGELOG 即可 |
| ✅ 老板无新决策 | 按原计划执行 |

## 3. 角色与仓库权限矩阵

| 角色 | 提案仓库 (mynotes/) | 编码仓库 (project/) | 使用的命令 |
|------|-------------------|-------------------|-----------|
| **DevMate** | ✅ 可写 | ✅ 可写 | acf-*, /zcf/* |
| **老板** | 👁️ 只读 | 👁️ 只读 | 无（人工评审） |
| **OpenCode** | ❌ 禁止写入 | ✅ 可写 | /zcf/arch-doc, /zcf/task-review |
| **Claude Code** | ❌ 禁止写入 | ✅ 可写 | /zcf:arch-doc, /zcf:task-review |

**关键规则**:
- ✅ 架构草稿**必须**先在提案仓库创建（DevMate 手工）
- ✅ 编码架构师**只能**写编码仓库
- ✅ 架构定稿后，DevMate 用 acf-sync 同步提案仓库 → 编码仓库

### 4.2 双层记忆结构（全局 + 项目）

#### 全局记忆（跨项目维度）

**位置**: `memory/YYYY-MM-DD.md`

**用途**: 记录当天所有项目的关键事件、决策、状态变更

**写入时机**: 任务接收后 1 分钟内

**格式**:
```markdown
# memory/YYYY-MM-DD.md

## In Progress

### [Project-A] 任务描述
- 需求澄清状态：✅ 已完成 / 🛑 待 Interview
- 架构评估：🟢 简单 / 🟡 复杂（需架构循环）
- 当前阶段：架构草稿 / 编码中 / 评审中
- 项目进展文件：`/workspace/Project-A/.acf/status/current-task.md`

### [Project-B] 任务描述
- ...
```

---

#### 项目记忆（单项目维度）⭐

**位置**: `<Project>/.acf/`

**用途**: 追踪项目级进展，支持重启后快速恢复

**目录结构**:
```
/workspace/<Project>/.acf/
├── status/
│   ├── current-task.md        # 当前任务状态（重启恢复用）
│   ├── current-phase.md       # 当前阶段概览
│   └── YYYY-MM-DD-status.md   # 每日状态报告（acf-status 生成）
├── temp/
│   ├── task-plans/            # 任务计划（phase*-tasks.md）
│   ├── arch-issues.md         # 架构问题记录
│   └── execution-log.md       # 执行日志
└── config/
    └── acf-triggers.yaml      # 触发器配置
```

**核心文件**: `current-task.md`（重启恢复的关键）

**格式**:
```markdown
# 当前任务状态

**项目**: <Project-Name>
**更新时间**: YYYY-MM-DD HH:MM:SS
**最后任务**: Task 007

## 任务进度

| 任务 ID | 状态 | 完成时间 | 评审状态 |
|---------|------|---------|---------|
| Task 001 | ✅ 完成 | 10:30 | ✅ 通过 |
| Task 002 | ✅ 完成 | 11:15 | ✅ 通过 |
| Task 003 | 🔄 进行中 | - | - |
| Task 004 | ⏳ 待执行 | - | - |

## 当前阶段

**阶段**: Phase 1 - MVP
**进度**: 2/10 (20%)
**阻塞点**: 无

## 架构状态

**架构文档**: ✅ 已定稿 (docs/architecture/2026-03-31-xxx.md)
**同步状态**: ✅ 最新 (SYNC-REPORT-2026-03-31.md)
**待评审 ADR**: 无

## 下一步

**立即执行**: Task 003 - 实现 XXX
**验收标准**: ...
**预计耗时**: 60 分钟
```

### 4.3 Gateway Restart 恢复检查

**收到 GatewayRestart 通知后，必须**:

#### 步骤 1: 读取全局记忆（了解当天整体情况）
```bash
cat memory/YYYY-MM-DD.md | grep -A 10 "## In Progress"
```

#### 步骤 2: 遍历活跃项目（恢复具体进展）
```bash
# 找到所有有 .acf 目录的项目
find /workspace -maxdepth 2 -name ".acf" -type d

# 对每个项目，读取当前任务状态
for project in /workspace/*/; do
    if [ -f "$project/.acf/status/current-task.md" ]; then
        echo "=== $(basename $project) ==="
        cat "$project/.acf/status/current-task.md"
    fi
done
```

#### 步骤 3: 检查决策树状态
- 是否卡在 Interview？→ 读取 `memory/YYYY-MM-DD.md` 的 Interview 记录
- 是否卡在架构评审？→ 读取 `mynotes/<Project>/docs/architecture/draft.md`
- 是否等待老板确认？→ 读取 `memory/YYYY-MM-DD.md` 的待确认事项

#### 步骤 4: 主动汇报
```markdown
## Gateway 重启恢复报告

**重启时间**: YYYY-MM-DD HH:MM
**重启原因**: {{原因}}

### 活跃项目进展

| 项目 | 当前任务 | 进度 | 状态 |
|------|---------|------|------|
| Project-A | Task 003 | 2/10 | 🔄 进行中 |
| Project-B | Task 001 | 0/5 | 🟢 就绪 |

### 待办事项
- [ ] Project-A: 继续 Task 003
- [ ] Project-B: 启动 Task 001
- [ ] 等待老板确认：xxx（如有）
```

### 4.4 项目进展更新规则

| 事件 | 更新文件 | 更新时机 |
|------|---------|---------|
| 任务开始 | `.acf/status/current-task.md` | 调用 acf-executor 前 |
| 任务完成 | `.acf/status/current-task.md` | 评审通过后 |
| 阶段完成 | `.acf/status/current-phase.md` | 所有任务完成后 |
| 架构变更 | `.acf/temp/arch-issues.md` | 发现问题时 |
| 每日站会 | `.acf/status/YYYY-MM-DD-status.md` | 每日 9:00（cron） |

**更新脚本**（计划添加）:
```bash
# .acf/scripts/update-status.sh
#!/bin/bash
PROJECT_PATH="${1:-.}"
TASK_ID="${2:-}"
STATUS="${3:-in_progress}"

cat > "$PROJECT_PATH/.acf/status/current-task.md" << EOF
# 当前任务状态
**更新时间**: $(date -Iseconds)
**当前任务**: $TASK_ID
**状态**: $STATUS
EOF
```

## 5. 违反决策树的后果与修复

| 违规行为 | 后果 | 修复方案 |
|---------|------|---------|
| 跳过 Interview 直接执行 | 需求理解偏差，返工 | 暂停任务 → 补 Interview → 更新记忆 |
| 未评估复杂度直接编码 | 架构缺陷，后期大改 | 暂停 → 补架构草稿 → 对比评审 |
| 编码架构师写提案仓库 | 仓库权限混乱 | 删除错误文件 → 更新 acf-triggers 阻止 |
| 未写入记忆文件 | Gateway Restart 后丢失上下文 | 立即补写 → 设置 cron 提醒 |
