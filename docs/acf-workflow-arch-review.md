## ❌ 设计遗漏项

### 遗漏 1: 编码循环任务流转规则（严重）

**问题**: 当前文档定义了"评审通过 → 继续下一个任务"，但**没有定义如何确定下一个任务**。

**现状**:
```bash
# 当前文档描述
/zcf:task-review "Task 001 完成"
# 评审通过 → 继续 Task 002  ← 如何确定 Task 002 是什么？
```

**影响**: 每次评审后需要人工决定下一步（如此次我问老板"选哪个"）

**修复方案**:

```markdown
## 3.2.1 任务流转规则（新增）

### 规则 1: 顺序执行（默认）
```
前提：implementation-plan.md 定义了任务依赖顺序
流转：Task N 完成 → 读取计划 → 执行 Task N+1
```

### 规则 2: 并行执行（独立任务）
```
前提：多个任务无依赖关系
流转：同时 spawn 多个执行者 Agent
```

### 规则 3: 阻塞等待（有依赖）
```
前提：Task N+1 依赖 Task N 的输出
流转：Task N 完成 → 标记 Task N+1 为"就绪" → 等待协调员调度
```

### 规则 4: 修复优先（P0 问题）
```
前提：评审发现 P0 级别问题（阻塞后续任务）
流转：暂停后续任务 → 创建修复任务 → 执行修复 → 恢复原计划
```

### 自动流转决策树
```
Task N 评审通过？
├─ 是 → 检查实施计划
│      ├─ 有下一个任务 → 自动执行 Task N+1
│      └─ 无下一个任务 → 阶段完成，触发阶段评审
│
├─ 轻微偏差 → 记录 CHANGELOG → 继续 Task N+1
│
└─ 严重偏差 → 暂停 → 通知架构师 → 等待架构调整
```
```

### 遗漏 2: 修复任务管理流程（中等）

**问题**: 评审报告生成了"修复清单"，但**没有定义修复任务的创建和执行流程**。

**现状**:
```markdown
## 📋 修复清单
### P0（阻塞 Task 002）
- [ ] 修复 Mock Fixture 错误（3 处）
```
→ 然后呢？谁修复？何时修复？如何验证？

**修复方案**:

```markdown
## 3.2.2 修复任务管理（新增）

### 修复任务创建
- 评审报告中的 P0/P1 问题 → 自动创建修复任务
- 任务命名：`Fix-XXX: [问题摘要]`
- 优先级：P0 > P1 > P2

### 修复任务执行
- P0 问题：阻塞后续任务，立即执行
- P1 问题：与正常任务交替执行（每 3 个正常任务后执行 1 个修复）
- P2 问题：阶段结束时统一处理

### 修复验证
- 修复完成后 → 重新运行相关测试
- 测试通过 → 关闭修复任务
- 测试失败 → 重新修复或升级问题级别
```

### 遗漏 3: 阶段/任务完成标准（中等）

**问题**: 没有明确定义"Task 完成"和"阶段完成"的验收标准。

**修复方案**:

```markdown
## 任务完成标准（新增）

### Task 完成定义（DoD）
- [ ] 代码实现完成
- [ ] 单元测试通过（覆盖率 ≥80%）
- [ ] 评审通过（无严重偏差）
- [ ] Git commit 完成
- [ ] 执行日志记录

### 阶段完成定义（DoD）
- [ ] 所有 Task 完成并通过评审
- [ ] 集成测试通过
- [ ] 阶段评审报告生成
- [ ] 技术债务汇总
- [ ] 下一阶段计划确认
```
