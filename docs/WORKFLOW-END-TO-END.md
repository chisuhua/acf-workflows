# 端到端工作流程文档

**版本**: v1.0  
**创建时间**: 2026-03-31  
**维护人**: DevMate

---

## 📖 文档目的

本文档清晰展示**老板的架构提案**如何在多方角色中流转，以及每个角色在工作流中的具体动作。

**目标读者**:
- **老板**: 了解你的提案如何被处理
- **DevMate**: 明确每个环节的操作职责
- **架构师/编码助手/评审员/分析师**: 理解自己在流程中的位置

---

## 🎭 角色总览

| 角色 | 身份 | 职责 | 输入 | 输出 |
|------|------|------|------|------|
| **老板** | 技术决策者 | 架构提案、技术选型审批 | 需求/想法 | 架构提案（mynotes） |
| **DevMate** | 技术合伙人/终端操作员 | 流程驱动、架构对比、决策执行 | 老板提案 | 任务分派、架构对比结论 |
| **架构师** | OpenCode (运行 arch-doc) | 架构设计、文档生成 | DevMate 命令 | 标准化架构文档 |
| **编码助手** | OpenCode (自主执行) | 代码实现、测试 | DevMate 任务 | 可运行代码 + 测试 |
| **评审员** | OpenCode (运行 task-review) | 任务评审、偏差检查 | DevMate 评审命令 | 评审报告 |
| **分析师** | OpenCode (运行 status) | 状态分析、进度报告 | DevMate 状态查询 | 进度报告 |

## 📝 详细步骤说明

### 阶段 0: 老板提案

**参与者**: 老板  
**输入**: 需求/想法  
**输出**: `mynotes/<Project>/docs/architecture/草稿.md`

**老板动作**:
```markdown
1. 在 mynotes 提案仓库创建架构草稿
   路径：mynotes/<Project>/docs/architecture/YYYY-MM-DD-草稿.md

2. 记录核心决策
   - 技术选型
   - 模块边界
   - 数据流向
   - 关键接口

3. 通知 DevMate
   "@DevMate 架构草稿已完成，请评审"
```

---

### 阶段 1: 架构循环（慢循环）

#### 1.1 DevMate 读取老板提案

**参与者**: DevMate  
**输入**: `mynotes/.../草稿.md`  
**输出**: 架构理解 + 待确认问题

**DevMate 动作**:
```bash
# 读取老板提案
cat mynotes/<Project>/docs/architecture/YYYY-MM-DD-草稿.md

# 记录待确认问题（如有）
# 写入：memory/YYYY-MM-DD.md
```

---

#### 1.2 DevMate → 架构师发送命令

**参与者**: DevMate + 架构师  
**输入**: 老板提案  
**输出**: OpenCode ACP session 启动

**DevMate 动作**:
```bash
# 方式 A: 通过 acf-executor
skill_use acf-executor task="/zcf/arch-doc '电商分析系统'" cwd="/workspace/ecommerce"

# 方式 B: 直接通过 ACP session 输入
# （在 OpenCode session 中输入）
/zcf/arch-doc "电商分析系统"
```

---

#### 1.3 架构师执行 arch-doc 工作流

**参与者**: 架构师（OpenCode 运行 arch-doc）  
**输入**: 任务 Prompt  
**输出**: `docs/architecture/YYYY-MM-DD-xxx.md`

**架构师动作**:
```
阶段 1: 研究
- 读取现有文档（docs/architecture/）
- 技术栈识别（package.json/pyproject.toml）
- 项目结构分析

阶段 2: 构思
- 生成 2-3 种架构方案
- 方案评估矩阵

阶段 3: 评审
- 完整性检查清单
- 架构一致性验证
- 输出标准化文档
```

**输出路径**:
```
docs/architecture/YYYY-MM-DD-<system-name>.md
docs/architecture/decisions/ADR-XXX-<title>.md
docs/architecture/reviews/YYYY-MM-DD-<system-name>-review.md
```

---

#### 1.4 DevMate 对比评审

**参与者**: DevMate  
**输入**: mynotes 提案 + 编码仓库标准化文档  
**输出**: 对比结论（一致/不一致）

**DevMate 动作**:
```bash
# 读取 mynotes 提案
cat mynotes/<Project>/docs/architecture/YYYY-MM-DD-草稿.md

# 读取编码仓库标准化文档
cat /workspace/<Project>/docs/architecture/YYYY-MM-DD-xxx.md

# 对比检查清单
- [ ] 架构文档结构一致
- [ ] 模块边界一致
- [ ] 接口定义一致
- [ ] 技术选型一致
- [ ] ADR 约束一致
```

---

#### 1.5 决策点：一致？

**参与者**: DevMate  
**输入**: 对比结论  
**输出**: 进入编码循环 / 返回架构循环

**DevMate 动作**:
```
一致 → 执行 acf-sync，进入阶段 2
不一致 → 通知架构师更新：
  /zcf/arch-doc "更新：XXX 模块接口定义"
  返回 1.2
```

---

### 阶段 2: 编码循环（快循环）

#### 2.1 DevMate 执行 acf-sync 同步

**参与者**: DevMate  
**输入**: 定稿架构文档  
**输出**: 同步后的编码仓库

**DevMate 动作**:
```bash
skill_use acf-sync
# 动作：提案仓库定稿文档 → 编码仓库
```

---

#### 2.2 DevMate → 编码助手发送任务

**参与者**: DevMate + 编码助手  
**输入**: 任务描述  
**输出**: OpenCode ACP session 启动

**DevMate 动作**:
```bash
skill_use acf-executor task="Task 001: 创建 Crawler 基类" cwd="/workspace/ecommerce"
```

---

#### 2.3 编码助手自主执行代码

**参与者**: 编码助手（OpenCode 自主执行）  
**输入**: 任务 Prompt + 架构文档  
**输出**: 可运行代码 + 测试 + Git 提交

**编码助手动作**:
```
1. 读取架构文档
   docs/architecture/YYYY-MM-DD-xxx.md
   docs/architecture/phases/phase-1/crawler/detailed-design.md

2. 创建 Git 分支
   git checkout -b feature/task-001-crawler-base main

3. 实现代码
   src/crawler/base.py
   src/crawler/exceptions.py

4. 编写测试
   tests/crawler/test_base.py
   tests/crawler/test_exceptions.py

5. 运行合规检查
   bash scripts/check-compliance.sh

6. 更新状态机
   EXECUTING → WAITING_REVIEW
```

---

#### 2.4 DevMate → 评审员发送评审命令

**参与者**: DevMate  
**输入**: 任务完成通知  
**输出**: 评审员角色唤醒

**DevMate 动作**:
```bash
# 通过 ACP session 输入（OpenCode）
/zcf/task-review "Task 001 完成"

# 或通过 ACP session 输入（Claude Code）
/zcf:task-review "Task 001 完成"
```

---

#### 2.5 评审员执行评审

**参与者**: 评审员（OpenCode 运行 task-review）  
**输入**: 评审命令  
**输出**: 评审报告（4 章节格式）

**评审员动作**:
```
一、任务执行结果
- 产出物清单
- Git 提交
- 测试结果 + 覆盖率
- 合规检查

二、架构一致性检查
- 一致项
- 偏差（轻微/中等/严重）

三、给 DevMate 的行动建议
- 需要确认的事项
- 需要执行的命令

四、下一步决策
- 下一任务
- 建议行动（选项 A/B/C）
```

---

#### 2.6 DevMate 查看评审报告

**参与者**: DevMate  
**输入**: 评审报告  
**输出**: 下一步决策

**DevMate 动作**:
```
查看报告 → 决策：
- 无问题 → 执行 acf-flow --next
- 轻微偏差 → 记录 CHANGELOG，继续
- 严重偏差 → 返回阶段 1（架构循环）
```

---

#### 2.7 DevMate → acf-flow 获取下一个任务

**参与者**: DevMate  
**输入**: 评审通过  
**输出**: Task 002

**DevMate 动作**:
```bash
skill_use acf-flow --next
```

---

### 阶段 3: 状态分析（按需）

#### 3.1 DevMate → 分析师发送状态查询

**参与者**: DevMate + 分析师  
**输入**: 状态查询命令  
**输出**: 分析师角色唤醒

**DevMate 动作**:
```bash
# 方式 A: 通过 acf-status
skill_use acf-status mode=full

# 方式 B: 通过 ACP session 输入
/zcf/status full
```

---

#### 3.2 分析师执行状态分析

**参与者**: 分析师（OpenCode 运行 status）  
**输入**: 状态查询命令  
**输出**: 进度报告

**分析师动作**:
```
- 架构文档完成度分析
- 模块设计完成度分析
- 任务完成度分析
- 效率指标计算
- 阻塞点识别
```

**输出格式**:
```markdown
# 项目状态报告

## 架构文档完成度
- 总体架构：✅ 已创建
- 错误处理策略：✅ 完整（>500 行）
- 测试策略：✅ 已创建
- 实施计划：✅ 已创建

## 模块设计完成度
- Phase 1: 5/5 模块完成
- Phase 2: 0/5 模块完成

## 任务完成度
- Task 001-005: ✅ 完成
- Task 006-010: 🔄 进行中

## 效率指标
- 本周完成任务：5 个
- 平均任务耗时：2.5 小时
- 测试覆盖率：85%

## 阻塞点
- 无 / XXX 模块等待架构评审
```

---

#### 3.3 DevMate 查看进度报告

**参与者**: DevMate  
**输入**: 进度报告  
**输出**: 调整决策

**DevMate 动作**:
```
查看报告 → 决策：
- 正常 → 继续执行
- 阻塞 → 调整资源/返回阶段 1
```

## 🎯 关键决策点

| 决策点 | 位置 | 决策者 | 决策依据 | 可能结果 |
|--------|------|--------|---------|---------|
| 架构对比 | 1.5 | DevMate | mynotes vs 编码仓库 | 一致→编码循环 / 不一致→架构更新 |
| 评审结论 | 2.6 | DevMate | 评审报告（偏差级别） | 无问题→继续 / 严重偏差→架构循环 |
